# 方案A详细前端设计 — Coze Vibe Coding 自定义页面

> 技术路线：通过 Coze Vibe Coding（自然语言生成前端页面）+ Coze 托管部署，开发量最低，自由度中等
>
> 认证方案：Coze 内置用户系统（首选，零开发）→ Cloudflare Access（企业场景）→ 自建登录+JWT（精细管理）

## 二、页面模块划分与功能职责

### 2.1 模块清单

| 模块 | 组件名 | 功能职责 | 数据来源 |
|------|--------|---------|---------|
| 顶部导航栏 | `TopNav` | 品牌标识、全局平台切换、素材库快捷入口、用户菜单 | 全局状态 |
| 登录页 | `LoginPage` | 用户身份认证（账号密码/手机验证码/第三方OAuth） | 认证服务 |
| 用户菜单 | `UserMenu` | 用户信息、积分余额、个人设置、退出登录 | 全局状态 |
| 左侧入口面板 | `EntryPanel` | 四大入口导航：对话/快速/自定义/素材库 | 全局路由 |
| 对话视图 | `ChatView` | 自由对话，Bot意图识别后自动串联流水线 | Coze Bot API |
| 快速生成视图 | `QuickGenView` | 4步引导式表单：品类→模板→信息→视频 | 模板库 + 用户输入 |
| 自定义生成视图 | `CustomGenView` | 节点式操作面板，逐步控制 | 各工作流API |
| 节点-图片生成 | `ImageGenNode` | 商品描述/模板选择→关键词补充→生成 | 图片生成工作流 |
| 节点-图片处理 | `ImageEditNode` | 去背景/换背景/调尺寸/加水印 | 图片编辑工作流 |
| 节点-视频生成 | `VideoGenNode` | 模板/自定义描述→生成视频 | 视频生成工作流 |
| 节点-保存收藏 | `SaveNode` | 下载/收藏Prompt/收藏脚本/存素材库 | 收藏工作流 |
| 素材库视图 | `MaterialView` | 历史素材列表、收藏模板、筛选/搜索 | 素材查询工作流 |
| 结果预览 | `ResultPreview` | 图片/视频预览、选图、下载、后续操作 | 生成结果 |
| 平台选择器 | `PlatformSelector` | 6大平台切换，显示当前平台规范 | 全局状态 |
| 模板卡片 | `TemplateCard` | 模板缩略图+名称+标签，点击选择 | 模板库 |
| 关键词标签 | `KeywordTags` | 平台热词展示，点击采纳/移除 | 关键词库 |
| 进度指示器 | `StepProgress` | 快速生成步骤进度 / 自定义生成节点进度 | 当前流程状态 |
| 状态栏 | `StatusBar` | 当前平台、任务状态、积分余额 | 全局状态 |
| Toast提示 | `Toast` | 操作成功/失败/警告的全局轻提示 | 全局事件 |
| 加载骨架屏 | `SkeletonLoader` | 图片/视频生成等待时的占位动画 | 组件内状态 |

## 三、核心交互流程

### 3.0 登录认证流程

```
用户访问应用
    │
    ▼
┌──────────────────────────────────────────┐
│  Coze 内置用户系统（默认方案，零开发）     │
│                                          │
│  · 访问应用URL → Coze自动检查登录状态    │
│  · 未登录 → 自动弹出Coze登录页           │
│  · 已登录 → 直接进入主应用               │
│  · 登录后 sys.user_id / sys.user_name    │
│    在页面和工作流中自动可用               │
│  · 用户对话/素材自动按 user_id 隔离      │
│                                          │
│  Coze登录页支持：                         │
│  · 手机号 + 验证码                       │
│  · 邮箱 + 密码                           │
│  · 第三方登录（按Coze配置）              │
└──────────────┬───────────────────────────┘
               │ 登录成功（Coze自动处理）
               ▼
┌──────────────────────────────────────────┐
│  进入主应用                                │
│                                          │
│  · 全局状态自动获取用户信息               │
│  · 导航栏显示用户头像+昵称               │
│  · 工作流调用自动携带 user_id            │
│  · 素材库按用户隔离                      │
│  · API调用时 user_id 作为会话标识        │
└──────────────────────────────────────────┘

── 如后续叠加 Cloudflare Access ──

┌──────────────────────────────────────────┐
│  双层认证流程                              │
│                                          │
│  用户访问 → CF拦截 → CF身份验证           │
│         → 通过 → Coze页面 → Coze用户登录  │
│         → 通过 → 主应用                   │
│                                          │
│  效果：先过企业安全网关，再过Coze用户识别  │
└──────────────────────────────────────────┘
```

## 九、认证系统设计

### 9.1 认证方案对比

| 方案 | 实现方式 | 适用场景 | 开发量 | 安全性 |
|------|---------|---------|--------|--------|
| **A. Coze内置用户系统** ⭐首选 | Bot发布时开启用户系统，Coze自动处理登录 | 所有场景，零开发首选 | **零** | 中（Coze托管） |
| **B. Cloudflare Access** | CF Zero Trust接管，用户访问时自动跳CF认证 | 团队内部/企业用户、需额外安全层 | 最低 | 高（CF企业级安全） |
| **C. 自建登录+JWT** | 前端登录页 → 后端签发JWT → 前端携带Token调用API | 需要精细用户管理、注册/套餐体系 | 中 | 中高 |
| **D. 第三方OAuth** | 微信/企业微信/飞书OAuth登录 | 企业用户、已有身份体系 | 低 | 高 |

**推荐策略**：先用方案A（零开发上线），后续按需叠加方案B/C。

### 9.2 方案A：Coze 内置用户系统（首选，零开发）

#### 核心机制

Coze 平台自带用户系统，在 Bot 发布配置中一键开启即可：

```
Coze 左侧「项目管理」 → 选择Bot/项目 → 发布 → Web SDK / Webapp 渠道
                                          │
                                          ▼
                                  用户系统设置：
                                  ✅ 开启 → 用户必须登录后才能使用
                                  ❌ 关闭 → 任何人可直接访问
```

> 默认建议先只用这一套。它最适合 MVP 和验证期，也和“尽量不自建后台”的路线一致。

#### 开启后的效果

| 特性 | 说明 |
|------|------|
| 自动登录页 | 访问应用时，Coze自动弹出登录页（手机号/邮箱） |
| 用户身份标识 | 系统自动为每个用户分配唯一 `user_id` |
| 数据隔离 | 不同用户的对话历史、素材库自动隔离 |
| 无需开发 | 零代码，发布配置中勾选即可 |

### 9.3 方案B：Cloudflare Access 接入

#### 架构

```
┌──────────┐     ┌──────────────────┐     ┌──────────────────┐
│  用户浏览器 │ ──→ │ Cloudflare Edge │ ──→ │ Coze 托管页面     │
│          │     │                  │     │ (origin server)  │
│          │     │ · Access Policy  │     │                  │
│          │     │ · 身份验证       │     │ · 前端页面        │
│          │     │ · JWT签发        │     │ · Bot/工作流      │
│          │     │ · 会话管理       │     │                  │
└──────────┘     └──────────────────┘     └──────────────────┘
       │                  │
       │    CF自动注入     │
       │  Cf-Access-JWT   │
       ▼                  ▼
  前端读取JWT解析用户信息
```

#### 配置步骤

| 步骤 | 操作 | 说明 |
|------|------|------|
| 1 | Cloudflare Dashboard → Zero Trust → Access → Applications | 创建应用 |
| 2 | 设置域名策略 | 将应用域名加入Access保护 |
| 3 | 配置身份提供商(IdP) | 支持邮箱OTP、Google、GitHub、企业IdP等 |
| 4 | 设置Access Policy | 定义谁可以访问（邮箱后缀/组/IP） |
| 5 | 前端读取用户信息 | 解析 `Cf-Access-Jwt-Assertion` Cookie |

#### 前端适配

```typescript
// 从Cloudflare Access JWT中解析用户信息
function parseCloudflareUser(): AuthUser | null {
  const jwt = getCookie('CF_Authorization');
  if (!jwt) return null;
  
  try {
    const payload = JSON.parse(atob(jwt.split('.')[1]));
    return {
      id: payload.sub,
      name: payload.name || payload.email,
      avatar: payload.picture || '',
      email: payload.email,
      loginMethod: 'cloudflare',
    };
  } catch {
    return null;
  }
}

// 检查是否通过CF认证
function isCloudflareAuthenticated(): boolean {
  return !!getCookie('CF_Authorization');
}
```

#### 优势

- **零代码**：无需开发登录页，CF自动拦截未认证请求
- **企业级安全**：CF全球网络安全防护
- **多种IdP**：支持邮箱OTP、Google、GitHub、企业SAML/OIDC
- **精细策略**：可按邮箱后缀、IP、组设置访问规则
- **审计日志**：CF自动记录所有访问日志

#### 局限

- 用户管理依赖CF，无法自建注册/套餐体系
- 免费版限50用户，付费版 $3/用户/月
- 登录页样式不可定制（CF统一页面）

### 9.6 分阶段实施策略

**推荐分阶段实施**：

```

│    - 双层保护，安全性更高
│
第三阶段：叠加自建认证（按需，+3天）
│
│  · 新增 LoginPage 注册/登录页面
│  · 对接 Supabase Auth 或自建认证后端
│  · 用户体系：注册/登录/套餐/积分管理
│  · 与Coze内置用户关联：
│    - 自建登录→获取JWT→调用Coze API时传user_id关联
│    - 三套用户可通过同一邮箱/user_id字段统一
│
第四阶段：独立部署+统一用户体系（可选）
│
│  · 自建认证为唯一入口
│  · 企业用户走 SAML/OIDC SSO
│  · 个人用户走 手机+验证码
│  · 完全脱离Coze内置用户体系
```

### 9.7 认证方案与套餐体系的关系

| 用户类型 | 认证方式 | 套餐 | 积分/月 |
|---------|---------|------|--------|
| 免费用户 | Coze内置登录 | 免费版 | 45,000 |
| 个人用户 | Coze内置登录(手机/邮箱) | 进阶版¥39.9 | 75,000 |
| 团队用户 | Coze内置 + Cloudflare Access | 高阶版¥99 | 144,000 |
| 企业用户 | SSO(SAML/OIDC) + Coze API关联 | 旗舰版¥199 | 244,000 |

> 这张表的重点是把“认证方式”和“套餐层级”对应起来，不代表所有层级都必须自建登录。MVP 直接用 Coze 内置登录就够了。

## 十、Vibe Coding 实现要点

### 10.1 Coze自定义页面能力

Coze Vibe Coding 支持通过自然语言描述生成自定义页面，关键能力：

| 能力 | 说明 |
|------|------|
| 页面渲染 | 自定义HTML/CSS/JS页面 |
| Bot对话嵌入 | 页面内嵌Bot对话组件 |
| 工作流调用 | 页面按钮/事件触发Coze工作流 |
| 知识库读取 | 页面可检索知识库内容 |
| 文件上传 | 支持图片上传到Coze |
| 多端适配 | 页面自动适配移动端 |

### 10.2 Vibe Coding Prompt 策略

由于Vibe Coding是通过自然语言让Coze生成页面，建议分模块描述：

**第一步：生成基础布局**
```
创建一个电商AI商图视频助手的应用页面：
- 用户系统：使用Coze内置用户系统（Bot发布时已开启登录），页面中通过 sys.user_id 和 sys.user_name 获取用户信息
- 顶部导航栏：Logo"电商AI商图视频助手"，右侧有平台选择下拉框、素材库按钮、用户头像下拉菜单
  用户菜单显示：用户名(sys.user_name)、积分余额、套餐等级、个人设置、退出登录
- 左侧面板240px宽：4个入口（对话、快速生成、自定义生成、素材库），点击切换右侧内容
- 右侧主内容区：根据左侧选择动态展示不同视图
- 底部状态栏：当前平台、任务状态、积分余额
```

**第二步：逐模块生成视图**
```
为快速生成创建4步表单视图：
Step1选平台(6个平台卡片)、Step2选品类+图片风格(带缩略图)、
Step3填商品信息+关键词推荐(可点击采纳)、Step4选视频模板(可选)，
底部[一键生成]按钮，顶部步骤进度条
```

**第三步：关联工作流**
```
"一键生成"按钮调用快速生成工作流，传入platform/category/style/product_name/product_info/video_style，
结果显示在ResultPreview组件中(3张图可选+视频播放器+操作按钮组)
```

### 10.3 组件复用策略

| 组件 | 复用场景 |

## 十二、方案A vs 方案B vs 方案C 对比

| 维度 | 方案A(Vibe Coding) | 方案B(Bot界面) | 方案C(独立前端) |
|------|-------------------|---------------|---------------|
| 开发量 | 低(3-4天) | 零(1-2天) | 中(8-12天) |
| UI自由度 | 中(受Vibe Coding能力限制) | 低(对话式) | 最高(完全自定义) |
| 交互体验 | 操作面板+对话，接近原生应用 | 纯对话，学习成本低但操作慢 | 最优 |
| 认证能力 | Coze内置(零开发)+CF/自建(可选叠加) | Coze内置用户系统 | 完全自定义 |
| 多端适配 | Coze自动适配 | Bot自动适配 | 需手动适配 |
| 部署 | Coze托管 | Coze托管 | 独立部署(Vercel等) |
| 维护成本 | 低(自然语言修改) | 最低(改Bot配置) | 中(代码维护) |
| 扩展性 | 中 | 低 | 最高 |
| 适合阶段 | 验证后需要更好体验 | 首次验证MVP | 长期正式运营 |

**结论**：方案A适合"Bot界面不够用但不想从头写前端"的阶段，在方案B验证需求后升级使用。
